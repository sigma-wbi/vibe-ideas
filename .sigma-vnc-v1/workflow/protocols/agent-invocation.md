# 동적 전문가 호출 프로토콜

> 참조: /qa, /ask
> 목적: 작업 성격에 맞는 전문가를 동적으로 구성하여 호출

---

## 동적 전문가 시스템

고정된 에이전트 대신, Claude가 작업 성격을 분석하여 **그때그때 적합한 전문가 역할을 직접 정의**하고 호출합니다.

각 전문가에 대해 다음을 동적으로 생성:
- **역할**: 전문 분야를 나타내는 역할 (예: "Docker 아키텍트", "쿼리 성능 분석가", "보안 감사관"). **가짜 인명(박진우, 김수현 등)을 절대 만들지 않는다.** 역할 자체가 이름이다.
- **전문 분야**: 해당 전문가가 다루는 영역
- **리뷰/분석 관점**: 구체적으로 무엇을 검토하는지

---

## 호출 조건

| 등급 | 전문가 팀 |
|------|---------|
| Patch | 없음 |
| Task | 선택적 (1-2명) |
| Feature | **권장** (2-3명) |
| Project | **권장** (3-5명) |
| Epic | **권장** (3-5명) |

---

## 호출 방식

### Task() 기반 병렬 호출

```
FOR each expert in panel:
    Task(
        subagent_type="general-purpose",
        prompt="당신은 '{역할}' 전문가입니다.
        전문 분야: {Claude가 작업 성격에 맞게 생성한 전문 분야 설명}
        리뷰 관점: {Claude가 작업 성격에 맞게 생성한 구체적 관점}

        ## 리뷰 대상
        {대상 내용}

        ## 출력 형식
        1. 핵심 의견 (2-3문장)
        2. 상세 분석
        3. 구체적 개선안 (코드 수준)"
    )
```

---

## 작업 성격별 패널 추천 (예시)

| 작업 성격 | 추천 패널 예시 |
|----------|--------------|
| API 설계 | API 아키텍트, 보안 감사관, DX 전문가, 성능 분석가, 도메인 전문가 |
| UI 구현 | UX 전문가, 접근성 감사관, 성능 최적화 전문가, 비주얼 QA |
| DB 스키마 | DBA, 데이터 무결성 전문가, 쿼리 성능 분석가, 보안 전문가 |
| 인프라 변경 | 클라우드 아키텍트, 보안 전문가, SRE, 비용 분석가 |
| 비즈니스 로직 | 도메인 전문가, 시스템 아키텍트, 테스트 전략가, 디버깅 전문가 |

> 위 테이블은 예시이며, 실제로는 작업에 최적화된 전문가가 동적으로 정의됩니다.

---

## 호출 프롬프트 구조

```markdown
# 역할: {역할}

당신은 {전문 분야} 전문가입니다.

## 리뷰 대상
{spec 문서 또는 코드 diff}

## 리뷰 관점
{Claude가 작업 성격에 맞게 동적 생성한 관점}

## 출력 형식
1. **핵심 의견** (2-3문장): 가장 중요한 평가
2. **상세 분석**: 항목별 분석
3. **구체적 개선안**: 실행 가능한 제안 (코드 수준)
```

---

## 결과 종합 (Orchestrator)

1. 각 전문가 결과 수집
2. 참여 전문가 명시:
   ```
   📋 리뷰 패널:
     1. {역할} — {전문 분야 한 줄}
     2. {역할} — {전문 분야 한 줄}
     ...
   ```
3. Side effect 분석 (개선안 간 상충 검출)
4. 기획의도 부합 여부 확인
5. 최종 분류: 채택 / 보류 / 기각
6. 리뷰 보고서 생성

## 실패 대응

|------|------|
