# Root Cause 분석 프로토콜 (5단계)

> 참조: /fix
> 목적: 5단계 에스컬레이션으로 문제의 근본 원인을 체계적으로 추적

---

## 5단계 분석 절차

### 1단계: 증상 수집

**목표**: 문제를 정확히 정의한다.

- 에러 메시지 전문 기록
- 재현 조건 파악 (항상/간헐적/특정 조건)
- 영향 범위 확인 (어떤 기능/사용자에 영향)
- 최근 변경 사항 확인 (`git log --oneline -10`)

**산출물**:
```
증상: {에러 메시지}
재현: {조건}
영향: {범위}
최근 변경: {관련 커밋}
```

### 2단계: 가설 수립

**목표**: 가능한 원인 후보를 나열한다.

- 최소 3개 가설 수립
- 각 가설에 확률 부여 (높/중/낮)
- 검증 방법 명시

**산출물**:
| # | 가설 | 확률 | 검증 방법 |
|---|------|------|----------|
| H1 | {{가설1}} | 높 | {{방법}} |
| H2 | {{가설2}} | 중 | {{방법}} |
| H3 | {{가설3}} | 낮 | {{방법}} |

### 3단계: 메모리 검색

**목표**: 유사 문제의 과거 경험을 활용한다.

```
memory_search(query="{에러 키워드}", type="gotcha", limit=5)
memory_search(query="{관련 기술}", type="pattern", limit=3)
```

- score >= 0.85: 가설에 반영 (우선순위 조정)
- 유사 gotcha 존재: 해결 방법 참조

### 4단계: 검증 (좁혀가기)

**목표**: 가설을 하나씩 검증하여 근본 원인을 특정한다.

- 확률 높은 가설부터 검증
- 검증 결과를 기록
- 기각된 가설도 사유와 함께 기록 (재발 방지)

**검증 방법**:
1. 로그 분석 (`Grep`, `Read`)
2. 코드 추적 (`project_query`)
3. 최소 재현 테스트
4. 변수 격리 (한 번에 하나씩 변경)

### 5단계: 수정 및 검증

**목표**: 근본 원인을 수정하고 재발 방지한다.

1. 근본 원인 수정 (증상이 아닌 원인)
2. 회귀 테스트 작성
3. 관련 gotcha 메모리 저장

**gotcha 저장 형식**:
```
상황: {어떤 상황에서}
문제: {어떤 문제가 발생}
원인: {근본 원인}
해결: {해결 방법 (원칙 수준)}
```

## /fix 에스컬레이션 (Hard Limit: 5회)

| 시도 | 행동 |
|------|------|
| 1회 | 1-5단계 1cycle |
| 2회 | 가설 재수립 + 범위 확대 |
| 3회 | 메모리 심층 검색 + 관련 파일 확대 |
| 4회 | AskUserQuestion (추가 정보 요청) |
| 5회 | "5회 시도 한도 도달. 수동 개입 필요." 보고 + gotcha 저장 |

---

## 금지 행동

- 증상만 가리는 워크어라운드 (근본 원인 미해결)
- 가설 없이 무작위 수정
- 에러 메시지 무시하고 추측으로 진행
- 같은 접근을 반복 시도
